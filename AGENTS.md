# 仓库开发约束

## 仓库位置

- HPC 地址：`/120090727/yutong/project/im`
- 本地地址：`/mnt/g/personal/im`
- 远程仓库：`https://github.com/petrichor20211/im.git`

执行命令或修改文件前，必须先确认当前所在环境和仓库路径，避免在错误副本中操作。

## 环境区别

- HPC/远端环境配有 GPU，涉及 CUDA、模型训练、GPU 推理、显存占用或 GPU 性能验证的任务必须在 HPC 上执行。
- 本地环境没有 GPU，只适合代码编辑、文档维护、数据检查以及不依赖 GPU 的轻量测试。
- 不得因本地缺少 GPU 而将 GPU 相关测试标记为已通过；应明确注明未在本地执行，并在必要时同步到 HPC 验证。
- 编写脚本时避免无条件假设 GPU 可用；若功能允许，应提供清晰的设备检测、参数配置或错误提示。

## 两端代码同步

- HPC 和本地是同一仓库的两个工作副本，必要时通过 Git 同步代码。
- 不要假定一端的未提交修改会自动出现在另一端，也不要用直接复制目录的方式代替 Git 同步。
- 同步前先执行 `git status --short --branch`，确认当前分支及工作区状态。
- 提交应保持目的单一、说明清晰；提交前检查 diff，避免带入缓存、密钥、生成文件或无关修改。
- 将一端的修改同步到另一端时，先在修改端提交并推送，再在目标端拉取。
- 拉取前必须处理目标端未提交的修改；不得直接覆盖或丢弃任何已有工作。
- 遇到分支分叉或合并冲突时，先检查双方提交和差异，不得擅自强制推送、重置或删除提交。
- 除非用户明确要求，不执行 `git push --force`、`git reset --hard`、`git clean -fd` 等破坏性操作。

## 长时间任务

- 模型训练、大规模数据生成、模型或依赖下载、长时间评测等预计耗时较高的任务，必须以后台方式运行，不得长期占用交互式命令或远程执行通道。
- 后台任务应使用 `nohup`、`setsid` 或调度系统启动，将标准输出和错误写入明确的日志文件，并保存 PID 或作业 ID；建议额外记录退出码到状态文件。
- 启动前应向用户说明将执行的任务、日志路径和预计资源开销；启动后立即返回 PID/作业 ID，不等待任务结束。
- 查看进度时使用 `ps`、调度器查询或短时间的 `tail`，不得通过持续前台跟踪日志阻塞对话。
- 后台任务尚未结束时，只能报告“运行中”；必须检查退出码、日志和预期产物后才能声明任务成功。

## 通用原则

- 仅修改当前任务所需的文件，避免无关重构。
- 优先遵循仓库已有的结构、文档、格式和测试约定。
- 修改完成后运行适用的检查或测试，并报告未能执行的项目。
- 不得将 API Key、Token、密码或其他敏感信息提交到仓库。
